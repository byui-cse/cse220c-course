Efficient Data Storage and Access

Beyond Arrays

Up to this point, the only type of container you've seen is a C-type array being used in C++. While these arrays are interesting, they lead to so many common coding crashes and security defects that the C++ documentation says that they are evil and should be avoided in almost every situation. This is why C++ has a set of standard containers. These include, but are not limited to, spans, pairs, tuples, vectors, lists, maps, and non-C arrays. The code for these containers is highly optimized. Rarely is anyone able to write more efficient code to store multiple instances of classes compared to these standard containers. This class focuses on lists and maps.

Consider the PharmaceuticalStockModel class you saw last week. It Has_A unique pointer to a C type array of Medications. 

class PharmaceuticalStockModel{
    private:
        unique_ptr<Medication[]> in_stock_medications;
        long medicationCount;
        long medicationCapacity;
    public:
    /*
     * constructor and class function declarations here
     */
 }

While this works, it uses that evil C-type array. It can definitely be improved upon for flexibility, usability, and security.

Consider this situation. Using your application's User Interface (UI), your customer wants to select the name of a medication and see a list of all the bottles of that medication, when they were recieved, when they expire, and how many pills, or if liquid how much liquid, remains in each bottle. This can be done using C-type arrays, but the code will be slow since you will have to do a lot of searching...comparing the name of each medication in the array with the selected name and building a temporary C-type array to hold pointers or references to, heaven forbid copies of, each of the matching medications. Wow! Slow and error prone.

Maps

All maps, including the one in the standard library of C++, std::map, are very good at looking stuff up. Each map is a group of keys and values in pairs. These key-value pairs are so common in programming and applications that you've been using them for a long time without knowing it. Think about your music player on your phone. It has lists of the songs, artists, and music genre on your phone. When you select a song to play, what is actually happening? After all, the display doesn't contain the actual song file. Instead it's the name of the song, musician, or genre. Let's say you select a song name. That name is a key that is used to look up the song file to play. The song file is the value. Key - song name: Value - song file.

'Wait a minute', you say. Doesn't the map just have an array inside? Isn't it going to look through that array just like I could? The answer to both of these questions is no. It is much more sophisticated than that and you'll learn more about this when you take the CSE212 Programming with Data Structures and CSE232 Designing Data Structures classes. For now, just believe there are computer scientists who have more knowledge than you do right now and map is designed by them to be significantly better than using a hidden array.

Maps are good at looking things up. Looking up things in C++'s std::map is O(log n) so let's modify the PharmaceuticalStockModel class to have a map of medications instead of a unique_ptr to an array of medications. When this is done, what before was 

class PharmaceuticalStockModel{
    private:
        unique_ptr<Medication[]> in_stock_medications;
        long medication_count;
        long medication_capacity;
    public:
    /*
     * constructor and class function declarations here
     */
 }

becomes 

class PharmaceuticalStockModel{
    using std::string;
    using std::map;
    private:
        map<string,Medication> in_stock_medications;
    public:
    /*
     * constructor and class function declarations here
     */
 }

Let's take a look at the lines that changed starting with the declaration of the in_stock_medications instance variable. The changed line states that there is a instance variable named in_stock_medications of type map where the key is a std::string and the value is a list of Medications. This is Sooo much simpler! Gone is the unique pointer. Gone is the C-type array. and gone are the medication_count and medication_capacity instance variables. Those last two were only there because a C-type array was used. They aren't needed anymore. Maps have infinite capacity and keep track of how many things are stored in them themselves. We don't have to do this separately like we had to when a C-type array was being used.

Notice in the declaration of the map, that there is some code that looks strange. <string,Medication>. This is where you declare the types for the keys and values. The C++ standard map stores the keys and values in a std::pair instance. The key is the first element of the pair and the value is the second element of the pair. The map declaration line in the previous code snippet tells the compiler that the map should hold pairs of type std::pair<string,Medication>.

This is all good. But there is one big problem. The C-type array could have many bottles of any specific type of medication. Maps use keys to look up a single value. Oh no...now we can only have one bottle of each type of medication in the pharamcy. Don't worry. There is a solution.

Lists

Lists are containers that are very good at keeping a group of items in a specific order. They are also good at adding and removing additional elements at the beginning or end of the list, O(1). Since you've learned about and used lists before, draw on that experience to understand what your reading next.

The pharmacy needs to be able to have multiple bottles of each type of medication. We know we can use a map with the name of the medication as a key to do a fast lookup. What if instead of looking up a single bottle the map could look up a list of bottles? Then the PharmaceuticalStockModel class would be fast and flexible. That's a good design! To make this happen the map needs a key of type string and a value that is a list of medications, list<Medication>. The complete, fast, flexible change is now, 

class PharmaceuticalStockModel{
    using std::string;
    using std::map;
    using std::list;
    private:
        map<string,list<Medication> in_stock_medications;
    public:
    /*
     * constructor and class function declarations here
     */
        void add_tablet_to_stock(const TabletMedication& medication_to_add);
 }

There is another benefit of this design. If bottles of medication are always added at the end of the list for their type of medication. The medication purchased longest ago will always be at the front of the list. That way, when a perscription for some amount of medication is recieved, the bottle purchased longest ago is the one that will be used to fill the perscription. That guarentees the stock of that medication will get turned over appropriately, assuming there is a corelation between the dates the medications were recieved and the expiration date of the medication in the bottles.

What, then, would the code for the PharmaceuticalStockModel instance function add_tablet_to_stock look like that adds a tablet-type medication to an existing list for that medication stored in the map? To be able to write code for this method, there is one more tool needed.

Iterators

iterate: to say or do again or again and again.

Iterators are instances of classes that let you move through a container of elements. Like Python lists, all C++ standard containers have one or more instance functions that return iterators. So for example, if you had the list of integers, list<int>ages{1,17,5,23}, you could get an iterator to the beginning of the list, the end of the list, or somewhere in between. But, unlike Python iterators, you have to use an operator in order to get the value out of the iterator.

Get the Beginning of the List

list<int>ages{1,17,5,23}
auto front_age_iterator = ages.begin();
cout << *front_age_iterator<<endl;//prints out 1

Get the End of the List

list<int>ages{1,17,5,23}
auto end_age_iterator = ages.end();
cout << *end_age_iterator<<endl;//prints out 23

Find Some Place in the List

list<int>ages{1,17,5,23}
auto front_age_iterator = ages.begin();
auto back_age_iterator = ages.end();
//find the location of 5 in the list...search from the beginning to the end
auto found_age_iterator = ages.find(ages.begin(),ages.end(),5);

Moving Using Iterators

To move to the next element of the list you use the ++ operator, a reflection of the sibling relationship C++ has with C. 

list<int>ages{1,17,5,23}
auto front_age_iterator = ages.begin();
cout << *front_age_iterator<<endl;//prints out 1
front_age_iterator++;
cout << *front_age_iterator<<endl;//prints out 17

Bringing it All Together

Now, combining a map that uses a string for its keys and a list of medications as its value, the addTabletToStockModel method can now be done. Compare these next sentences with the code example below.

First, use the map's find method to get an iterator that contains the list of medications with the name of the medication as the key. The iterator for this map will have a key-value pair std::pair<string, list<Medication>> internally. Since the list is the second element of the pair, use the second property of the pair to access the list containing all the bottles of medication. Lastly, use the list's push_back method to add the medication to the end of the list...and you're done. The medication has been added. 

void PharmaceuticalStockModel::add_tablet_to_stock(const TabletMedication& medication_to_add){
auto found_medication_iterator = in_stock_medications.find(medication_to_add.get_name());
found_medication_iterator->second.push_back(medication_to_add);
}
